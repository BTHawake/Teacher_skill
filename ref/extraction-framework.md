# 统一提取框架

## 为什么需要统一提取

teacher-skill 有多个信息来源——用户提供的资料、开源 skill、网络搜索结果、现有课程。以前每个来源走不同的路径，提取质量参差不齐。

这个框架让**所有来源走同一条管道**，产出同一质量标准的教学模块。

所有来源归一化为三种输入类型：

| 输入类型 | 例子 | 处理方式 |
|---------|------|---------|
| 文本资料 | 用户上传的 PDF/文档、网页文章、粘贴的文字 | 直接阅读分析 |
| 开源 skill | 从 awesome-persona-skills 找到的 SKILL.md | 用 `reverse_distill.py` 先解析，再按四维提取 |
| 搜索结果 | 网络搜索返回的文章/页面 | 阅读摘要或全文后提取 |

---

## Step 1：来源归一化

无论什么来源，先统一为"文本池"：

```
用户资料（PDF/TXT/链接） ──→ 文本化
开源 skill ──→ frontmatter + 心智模型段落
网络搜索 ──→ 内容摘要 + 原文链接
            │
            ▼
        统一文本池（所有内容集中在一起）
```

### 来源优先级

当同一观点有多个来源时，优先级排序：

| 优先级 | 来源类型 | 说明 |
|--------|---------|------|
| P0 | 用户自己提供的资料 | 用户最信任的信息 |
| P1 | 开源 skill 中的一手来源 | skill 中标注了来源的引用 |
| P2 | 开源 skill 中的内容（无来源） | 可用但需要交叉验证 |
| P3 | 网络搜索结果 | 搜索到的公开信息 |
| P4 | AI 自身知识 | 训练数据中的信息（最低优先级，需标注"基于 AI 知识"） |

> 使用原则：有 P0 不用 P1，有 P1 不用 P2，以此类推。只有当高优先级来源的信息不足以支撑完整课程时，才使用低优先级来源。

---

## Step 2：四维提取

从归一化的文本池中，按四个维度提取可教知识点：

### 维度①：核心观点

此人反复强调的核心理念、思维框架、世界观。

**提取方法**：
- 在所有来源中搜索此人的核心主张（"他反复说的是什么？"）
- 同一个观点在 ≥2 个独立场景/来源中出现 → 标记为核心观点
- 只在一个来源中出现 → 标记为"单源观点"，需要验证

**输出格式**：
```
观点：[一句话描述]
来源：[URL或资料名称]
出现频次：[在几个独立来源中出现]
适用场景：[这个观点在什么情况下最有用]
```

### 维度②：实际应用

这些核心观点在实际问题中怎么用。

**提取方法**：
- 从案例/故事/对话中提取"具体是怎么操作的"
- 关注"他做了什么"而不是"他说了什么"
- 每个应用需要有具体的情境描述

**输出格式**：
```
概念：[对应的核心观点]
应用场景：[具体情境描述]
做法：[他实际是怎么做的]
结果：[发生了什么]
是否可迁移：[这个做法能用在别处吗？]
```

### 维度③：多视角

他人对此人的评价、批评、补充视角。

**提取方法**：
- 搜索"XX的批评""XX的局限""XX的争议"
- 关注不同立场的评价（不只看赞美）
- 每个视角标注来源和立场

**输出格式**：
```
视角：[一句话总结]
来源：[URL]
立场：[称赞/批评/中立]
价值：[这个视角补充了什么盲点]
```

### 维度④：案例库

可用于教学的生动例子、故事、矛盾点。

**提取方法**：
- 搜索此人最著名的轶事、决策案例、争议事件
- 优先选择"有细节、有转折、有结果"的故事
- 标注每个案例对应的核心观点

**输出格式**：
```
案例：[标题]
对应观点：[关联的核心观点]
详情：[故事/事件的经过]
教学用途：[适合用来解释什么概念]
```

## Step 3：三重验证（质量门）

四维提取出的每一个"候选知识点"（包括核心观点、应用、案例），都必须通过三重验证才能进入教学模块：

### 第一重：跨域复现

这个知识点在 ≥2 个不同场景/来源中出现过吗？

```
通过条件：在 ≥2 个独立来源中出现
不通过：只在一个来源中出现 → 标记为"单源观点"，不进入核心模块

例外：如果该来源是用户提供的资料（P0），
      且同一观点在资料中反复出现≥3次 → 可通过
```

### 第二重：生成力

这个知识点能用来推断此人面对新问题的立场吗？

```
通过条件：能回答"如果此人遇到[新场景]，他会怎么用这个知识点？"
不通过：无法用知识点推导出新场景的结论 → 降级为补充信息
```

### 第三重：排他性

这个知识点是此人独特的方式，还是放之四海皆准的常识？

```
通过条件：换个其他人，不会用同样的方式表达同一个观点
不通过：换了谁都会这么说 → 降级为背景知识，不单独作为教学模块

示例：
- "不要骗自己"（费曼）—— 排他性高 → 通过 ✓
- "要努力工作" —— 排他性低 → 降级 ✗
```

### 验证结果处理

| 验证结果 | 处理方式 |
|---------|---------|
| 三重通过 | 作为独立教学模块 |
| 仅 1-2 重通过 | 降级为模块内的补充材料/案例/拓展阅读 |
| 零重通过 | 丢弃 |

---

## Step 4：课程组装

通过验证的知识点按以下方式组装为教学课程：

```
通过的知识点集合
    │
    ├─ 按关联度分组（相关的观点放一起）
    ├─ 按难易度排序（从简单到复杂）
    ├─ 每个组 = 一个教学模块
    ├─ 每组配 1-2 个案例（从案例库中取）
    └─ 诚实边界：根据未通过的知识点 + 来源质量自动生成
    │
    ▼
  课程文件（subjects/xxx-teacher-SKILL.md）
```

### 模块数量控制

| 总知识点数 | 建议模块数 | 说明 |
|-----------|-----------|------|
| 1-3 个 | 2-3 个 | 知识少时做成微型课程 |
| 4-6 个 | 3-5 个 | 标准课程 |
| 7-10 个 | 5-7 个 | 深度课程 |
| >10 个 | 选 top 7 | 宁缺毋滥，多余降级为拓展阅读 |

### 来源标注

每个模块必须标注信息来源：

```
## 模块 2：[观点名]

### 概念讲解
[核心观点内容]

> 来源：[URL/资料名]

### 案例
[案例详情]
> 来源：[URL/资料名]
```

## 完整流程一览

```
用户输入（任意形式）
    │
    ▼
Step 1：来源归一化（统一为文本池，按 P0-P4 排序）
    │
    ▼
Step 2：四维提取（核心观点/实际应用/多视角/案例库）
    │
    ▼
Step 3：三重验证（跨域复现→生成力→排他性）
    │
    ▼
Step 4：课程组装（分组→排序→配案例→标注来源）
    │
    ▼
进入 teacher-skill 教学流程（Phase 1→2→3→4→5）
```
