# 教学效果验证框架

## 为什么要做验证

"我懂了" = 自欺高发区（反自欺模型1指出这一点）。用户说理解不代表真的理解。
客观验证题的设计目的是：**用一个用户无法通过"听起来合理"来糊弄的方式，确认理解是否真实。**

## 验证题分类

每节课/每个阶段结束后，AI 从以下三类题中选择 1 道出给用户：

### 类型 A：概念辨别题（最低难度）

用户需要在多个选项中选出符合/不符合概念的正确描述。

**模板：**
```
关于 [概念]，下面哪个说法是错的？
A. [正确说法]
B. [正确说法]
C. [错误说法 — 伪装成正确的常见误解]
D. [正确说法]
```

> 使用时，把 A/B/C/D 的顺序打乱，不要让正确答案总是同一个位置。
> 用户的选项和评语记录到学习状态中。

**记录到学习状态**：记录用户选择的答案（字母），以及正确/错误。

### 类型 B：应用迁移题（中等难度）

给用户一个**新的场景**——不是教学中用过的例子——让用户应用刚学的概念。

**模板：**
```
我们之前用 [例子A] 解释了 [概念]。现在给你一个新的场景：
[新场景描述]

在这个场景中，[概念] 是怎么体现的？请用 2-3 句话说明。
```

> 关键：场景必须是全新的，不是教学中用过的例子。否则用户在"回忆"而不是"应用"。

**评分标准**：
- 正确应用 + 能说明原因 → 通过
- 部分正确但有混淆 → 需要回顾
- 完全错了或说"不知道" → 需要重新教学

**记录到学习状态（JSON格式）**：
```json
{
  "question_type": "application",
  "question": "新场景的问题",
  "user_answer": "用户的回答",
  "result": "pass | partial | fail",
  "notes": "AI的评判依据"
}
```

### 类型 C：教授验证题（最高难度，推荐中层以上用户）

费曼学习法的核心：让用户教 AI。

**模板：**
```
假装你是一个老师，我（AI）是你的学生。
请用你的话向我解释 [概念]，让我能真正理解它。
我在过程中会提问，你要能回答我的追问。
```

> 这是一种实时验证——AI 在听用户解释时，可以（也应该）追问。如果用户在追问下能自圆其说，才算真正理解。

**评分标准**：
- 能用大白话解释 + 能回答追问 → 通过
- 能解释但追问后暴露模糊 → 需要回顾
- 解释不了或依赖术语 → 需要重新教学

**记录到学习状态（JSON格式）**：
```json
{
  "question_type": "teach_back",
  "user_explanation": "用户解释的摘要",
  "ai_followups": ["追问1", "追问2"],
  "user_responses": ["回答1摘要", "回答2摘要"],
  "result": "pass | partial | fail",
  "notes": "AI的评判依据"
}
```

## 阶段验证与最终验证

### 阶段验证（Phase 4）

每完成 3-5 个单元后，出 1 道综合题：

- 如果是弱基础用户：类型 A（概念辨别）
- 如果是中等基础用户：类型 B（应用迁移）
- 如果是强基础用户：类型 C（教授验证）

验证结果通过 `scripts/learning_state.py --update` 存入学习状态的 `verification.checkpoints`。

### 最终验证（Phase 5）

全部课程结束后，出 1-2 道综合性题目。推荐类型 C（教授验证）。

最终验证结果存入 `verification.final_result`。

## 验证驱动的教学调整

| 验证结果 | 处理方式 |
|---------|---------|
| 通过（pass） | 继续下一个阶段/单元 |
| 部分通过（partial） | 回顾对应单元中模糊的知识点，再出 1 道同类题确认 |
| 未通过（fail） | 回到对应单元，换一种方式重新教学（用不同的类比、不同的例子），重新出验证题 |

> 重要：fail 后回到教学时，必须换一种解释方式。不能用同样的内容再讲一遍——那只是"再读一遍笔记"，不是"再理解"。

## 类型 D：思维模型质量验证（v2.2 新增）

当教学涉及**思维模型/心智模型**（如费曼的反自欺、巴菲特的护城河），可用以下三阶验证法确认用户是否真正掌握了该模型：

### 第一阶：跨域复现（Cross-domain Recurrence）

用户能否在 **2 个以上不同领域** 中识别出同一个模型？

"我们在投资案例中讲了'反脆弱'这个概念。现在看这个例子：一个创业公司故意让团队保持小规模，不融资——这是反脆弱吗？为什么？"

> 如果用户只能在一个领域（教学时用的例子）中识别模型 → 可能只是记住了名字，不是真正理解

### 第二阶：生成力（Generative Power）

用户能否用这个模型**推断**该人物对**新问题**的立场？

"如果费曼活到了 AI 时代，你觉得他会怎么看待'AI 写论文'这个问题？用他的哪个模型来支持你的判断？"

> 如果用户无法将模型应用到新场景 → 模型尚未内化

### 第三阶：排他性（Exclusivity）

用户能否区分**这个人的模型 vs 另一个人的模型**？

"巴菲特和费曼都强调'诚实'。但巴菲特的'诚实'和费曼的'不自欺'是一回事吗？区别在哪里？"

> 如果用户把两个不同思想家的相似概念混为一谈 → 只记住了表面，没理解内核

## 与学习状态的集成

验证结果通过以下格式更新学习状态：

**阶段验证记录格式：**
```json
{
  "checkpoints": [
    {
      "phase": 1,
      "unit": "单元名称",
      "question_type": "application",
      "result": "pass",
      "score": 85,
      "weak_areas": [],
      "timestamp": "2026-06-04T..."
    }
  ]
}
```

**最终验证记录格式：**
```json
{
  "final_result": {
    "passed": true,
    "overall_score": 90,
    "strong_areas": ["概念A", "概念B"],
    "weak_areas": ["概念C"],
    "recommendation": "建议继续学习相关话题...",
    "timestamp": "2026-06-04T..."
  }
}
```

## 学前 vs 学后对比验证（可选）

如果用户愿意，可以在学前一节课开始前做一次"摸底测试"（1 道类型 A），学完后再做同一道题（换选项顺序），对比两次答案的变化。这是最直观的"我确实学到了"的证据。

```
教学前摸底:
  用户对 [概念X] 的认知: [记录]
  
教学后验证:
  用户对 [概念X] 的认知: [记录]
  
进步: [AI 总结用户的变化]
```
