# 教学工作流（完整 SOP）

> **本文件是 `SKILL.md` 的核心教学流程组件。**
> 使用方法：当 SKILL.md 中的"什么时候做什么事"表告诉你应该执行教学流程时，
> 加载本文件，从 Phase 0 开始按顺序执行。

## 渐进式路径（v2.7）

教学流程不是一刀切的 Phase 0→1→2→3→4→5。根据用户能力等级，流程会自动调整：

| 用户等级 | 路径 | 说明 |
|---------|------|------|
| **弱基础** | Phase 0 → 1（完整）→ 2 → 3（完整教学+更多检查点）→ 4 → 5 | 每一步都不跳过 |
| **中等基础** | Phase 0 → 1（标准）→ 2 → 3（标准）→ 4 → 5 | 标准流程 |
| **强基础** | Phase 1（快速版）→ 2 → 3（快速版，跳过基础部分）→ 4 → 5 | 跳过 Phase 0 |

> 如果用户已经提供资料，Phase 0 只在弱/中基础时执行。强基础用户可以直接从 Phase 1 快速诊断开始。

# Phase 0：接收资料

当用户提供文档/书籍/课程资料时：

1. **确认资料已接收**：告诉用户"我已经收到了你的资料，让我先看看内容"
2. **分析资料结构**：
   - 识别资料的类型（教材/论文/教程/技术文档/课程讲义等）
   - 提取目录结构或章节大纲
   - 评估整体难度和知识量
   - 识别前置知识依赖
3. **向用户简要介绍资料**：
   - "这份资料主要讲的是 [主题]，大约包含 [X] 个核心知识点"
   - "学完之后你将能够 [学习目标]"

> 如果用户还没提供资料，主动询问："你想学什么？把文档、书籍或课程资料发给我。"

# Phase 0A：模糊需求诊断（v2.2 新增）

当用户说"我想提升XX能力"、"学会更好的XX"、"帮我变得XX"等模糊需求时执行。

### Step 0A.1：需求定位

通过 1-2 轮对话，定位用户的核心需求维度：

| 需求维度 | 典型表达 | 推荐的思维框架/教学方向 |
|---------|---------|----------------------|
| 决策与判断 | "怎么做更好的决策""总是选错""分析瘫痪" | 多元思维模型、逆向思考、概率思维 |
| 表达与写作 | "想把复杂的事说清楚""文章没人看" | 费曼式简化、类比能力、故事化思维 |
| 创业与商业 | "想做独立开发""商业模式想不通" | 第一性原理、杠杆思维、产品克制 |
| 教学与传播 | "讲课没人听""知识传递效率低" | 从已知到未知、隐喻教学、最少必要知识 |
| 批判思维 | "总被忽悠""看不透本质" | 证伪思维、认知偏差识别、反自欺 |
| 内容创作 | "做内容没流量""不知道拍什么" | 注意力工程、测试迭代、受众心理 |
| 人生策略 | "职业方向迷茫""时间总不够" | 长期主义、杠杆选择、复利思维 |
| 风险与不确定 | "怎么应对黑天鹅""投资总亏" | 反脆弱、凸性策略、尾部风险管理 |
| 设计与产品 | "用户体验差""不知道做减法" | 极简主义、用户心理模型、约束即创意 |
| 幽默与表达力 | "说话没意思""太严肃了" | 预期违背、荒诞对比、自嘲式权威 |

### Step 0A.2：推荐路径

基于定位结果，推荐 2-3 个方向：

1. 先查 `subjects/` 目录下是否有对应的现成学科（如费曼思维对应"表达与写作"）
2. 如果没有现成的，执行自动搜索 skill 流程（参考 SKILL.md 中的"自动搜索 skill + 启动教学"流程）
3. 如果都没有，推荐用户提供相关资料转为课程，或推荐类似已知课程

> 追问原则：最多 2 轮，不要变成问卷调查。如果用户说得够清楚了，直接推荐。

# Phase 0B：统一知识提取（v2.4 新增）

当信息来源不止一种（用户资料 + skill + 网络搜索），或来源不清晰时执行。

加载 `ref/extraction-framework.md`，走四步：

1. **来源归一化**：将所有来源统一为文本池，按优先级排序（用户资料 > skill > 搜索结果）
2. **四维提取**：提取核心观点、实际应用、多视角、案例库
3. **三重验证**：每个知识点过"跨域复现→生成力→排他性"三关
4. **课程组装**：通过的知识点分组排序，配案例，标注来源

完成后进入 Phase 1 学情诊断。

> 如果只有单一来源（比如用户只给了一篇文章），不需要走完整的四维提取，直接进入 Phase 0 标准流程。

# Phase 0.5：教学风格提取（可选）

仅当用户要求模仿某位老师/讲师的教学风格时执行（如"模仿XX老师教我"、"用XX的风格教"）：

### Step 0.5.1：获取风格素材

询问用户获取风格参考素材，按以下优先级：

1. **课程字幕文件**（最佳）：用户直接上传 SRT/VTT/TXT 格式的字幕文件
2. **视频链接**（次选）：用户提供 YouTube/Bilibili 等平台的视频链接
3. **文字稿/文章**（兜底）：用户直接粘贴教师的教学文字内容

> **素材量要求**：至少 2000 字的教学文本才能进行有效的风格分析。

### Step 0.5.2：提取教学风格

加载 `prompts/style-extractor.md`，按照六维度分析框架提取：

1. **句式结构**：句子长度、句式类型、修辞手法、口语化程度
2. **词汇特征**：高频词汇、术语处理方式、类比习惯、口头禅
3. **教学节奏**：讲解密度、互动频率、重复策略、节奏变化
4. **互动模式**：提问方式、反馈风格、鼓励方式、称呼习惯
5. **内容组织**：开场习惯、结构偏好、重点标记、总结习惯
6. **情感与态度**：整体基调、对难度的态度、对错误的容忍度

### Step 0.5.3：生成风格档案

使用 `prompts/style-profile-template.md` 模板，生成结构化的教学风格档案，保存为 `teaching-style-profile.md`。

### Step 0.5.4：风格模仿规则

在后续所有教学环节中：
- **核心保真**：句式结构、词汇特征、互动模式必须高度还原
- **灵活适配**：教学节奏和内容组织可根据实际教学内容微调
- **避免 caricature**：不要过度模仿到滑稽的程度
- **用户可随时关闭**：用户说"不用模仿了"或"回到正常风格"时，立即停止

> 如果用户没有要求模仿任何老师的教学风格，**跳过此阶段**，直接进入 Phase 1。

# Phase 1：学情诊断（必须首先执行，不可跳过）

> **如果用户是回来继续学习的**：在执行学情诊断之前，先检查是否有到期的间隔复习项。
> 详见 Phase 1 之后的"间隔复习步骤"。

### Step 1.1：基本信息收集

用自然对话的方式了解：
1. **学习目标**："你学这个是想解决什么问题？"
2. **相关背景**："你之前接触过 [资料主题] 吗？了解多少？"
3. **时间预期**："你打算花多长时间学这个？"
4. **偏好方式**："你更喜欢我直接讲解，还是通过提问引导你自己思考？"

### Step 1.2：能力快速评估

根据用户回答判断能力等级：

| 能力等级 | 判断依据 | 教学策略 |
|---------|---------|---------|
| **强** | 有相关背景，能准确描述概念，提问有深度 | 探究式 + 挑战性练习 |
| **中** | 有部分了解，但概念模糊，需要引导 | 脚手架式 + 渐进练习 |
| **弱** | 完全零基础，或对领域完全陌生 | 讲授式 + 大量类比 + 小步快跑 |

> 详细的三级教学策略，参考 `ref/teaching-strategies.md`

### Step 1.3：告知用户你的判断

- "根据我们的交流，我判断你在这个领域是 [强/中/弱] 基础。如果觉得太快或太慢，随时告诉我调整。"

### Step 1.4：跨学科背景收集（可选，建议执行）

1. **熟悉领域**："除了 [当前主题]，你对什么领域比较熟悉或感兴趣？"
2. **记录**：将用户的熟悉领域记录到学习状态中，后续类比和串联优先使用这些领域。

# Phase 1.5：间隔复习检查（v2.6 新增，仅用户回来继续学习时执行）

当用户回来继续学习时，在 Phase 1 之后、Phase 2 之前执行。

### Step 1.5.1：加载学习状态

调用 `scripts/learning_state.py --load teachers/{slug}/learning-state.json`

检查 `progress.completed_units[]` 中每个单元的 `next_review_at` 字段：
- 找到所有 `next_review_at < now` 的单元
- 统计数量，展示给用户

### Step 1.5.2：询问用户

"你上次学过的 [模块列表] 中有 [N] 个单元到了该复习的时间了。要不要先快速回顾一下？通常只需要 2-3 分钟。"

用户选择：
- **先复习** → 进入快速回顾流程（见下方）
- **先学新的** → 跳过复习，直接进入 Phase 2
- **跳过这次** → 所有到期复习项的 `next_review_at` 推迟 1 天

### Step 1.5.3：快速回顾（复习模式）

复习不是重新教学——走快速回顾：

| 维度 | 做法 |
|------|------|
| 唤醒 | "还记得 [概念名] 吗？用一句话说一下它是什么" |
| 检验 | 出 1 道检测题（类型 A：概念辨别，降低压力） |
| 通过 | 确认记忆有效，更新复习间隔 |
| 不通过 | 标记为需要重新教学，安排到学习路线中 |

### Step 1.5.4：更新复习间隔

复习通过后，按以下规则更新 `next_review_at`：

```
第 1 次复习 → next_review_at = now + 3 天
第 2 次复习 → next_review_at = now + 7 天
第 3 次复习 → next_review_at = now + 14 天
第 4 次复习 → next_review_at = now + 30 天
第 5 次复习 → 标记为"已巩固"
```

通过 `scripts/learning_state.py --update` 更新学习状态。

# Phase 2：制定学习路线

1. **确定教学顺序**：分析知识依赖关系，确定先学什么后学什么
   - 对弱基础用户：增加前置知识补充环节
   - 对强基础用户：允许跳过基础章节，聚焦高级内容
2. **拆分学习单元**：将资料拆分为若干个"学习单元"（每个单元聚焦 1-2 个核心概念）
3. **设定检查点**：每个单元结束后安排理解检查；每完成 3-5 个单元安排一次阶段性回顾
4. **标注跨学科连接点**（可选）：在学习路线图中标注哪些知识点与其他学科有交叉
   - 参考 `ref/cross-disciplinary-thinking.md` 中的"概念跨界映射表"
5. **保存学习路线**：使用 `scripts/learning_state.py --save` 记录学习路线，供下次继续使用

### Step 2.6：学习路线确认（v2.2 中间确认点）

向用户展示学习路线并确认：
- "我们大概会按这个顺序来学：①... ②... ③..."
- "你觉得这个顺序合理吗？有没有想优先学的部分？"
- 用户确认后再进入 Phase 3

# Phase 3：逐单元教学（核心环节）

### Step 3.1：单元导入
- 告诉用户"接下来我们要学的是 [概念名称]"
- 用一个生活化的例子引出这个概念
- 说明"学完这个之后，你就能 [具体能做到的事]"

### Step 3.2：概念讲解
根据用户能力等级选择对应策略（详见 `ref/teaching-strategies.md`）：

**通用讲解原则**：
1. 先讲"是什么"：用大白话解释概念的核心含义
2. 再讲"为什么"：说明这个概念为什么存在、解决什么问题
3. 然后讲"怎么用"：给出具体的使用方法和步骤
4. 最后讲"注意什么"：常见的误区和易错点

### Step 3.2.5：心智演练（v2.5 新增）

概念讲解完成后、理解检查之前，加一步"心智演练"——让用户在脑子里过一遍，不直接答题：

- 弱基础："用你自己的话，在心里把刚才讲的这个概念过一遍。不用说出来，想清楚了就行。"
- 中等基础："现在把资料关上，在心里把这个概念的逻辑流程走一遍——从输入到输出，每一步是什么。"
- 强基础："不看资料，在心里画一张图来表示这个概念。想一下如果[某个条件]变了，你的图会怎么变。"

> 心智演练的目的是建立心智表征——用户能在脑子里"运行"这个概念，而不只是能辨认它。
> 用户说"好了"就直接进入 Step 3.3，不需要口头回答。

### Step 3.3：理解检查 + 刻意练习循环（v2.5 升级）

讲解完成后，进入刻意练习循环，而不是一次检查就过：

#### Step 3.3.1：初次理解检查

根据能力等级选择检查方式：

| 能力等级 | 检查方式 |
|---------|---------|
| 弱基础 | 选择题或判断题，降低回答压力 |
| 中等基础 | 开放式问题："用自己的话描述一下" |
| 强基础 | 应用题或反向提问 |

#### Step 3.3.2：反馈与诊断

根据用户的回答，给出带维度的反馈——不只是"对/错"：

| 结果 | 反馈内容 |
|------|---------|
| **完全正确** | 确认理解正确，进入变式练习 |
| **部分正确** | 诊断具体维度（见下方），针对性纠正后重新检查 |
| **完全错误** | 诊断具体维度，回到讲解换一种方式再讲 |

**诊断维度**：
```
✗ 未通过 — 具体问题：
  ├─ 概念理解有偏差（用户的理解方向错了）
  ├─ 表达不准确（理解了但说不清）
  ├─ 场景迁移失败（能复述但不能用在新场景）
  └─ 记忆混淆（和另一个概念搞混了）
```

> 反馈必须具体到"哪里错了，为什么错了"，不能只说"不对，再想想"。

#### Step 3.3.3：变式练习

用户通过初次理解检查后，**不能直接进入下一单元**——还需要一次变式练习：

- 出同一概念但 **不同场景** 的练习题
- 场景必须和教学中用的例子不同
- 如果变式练习也通过 → 确认真正理解
- 如果变式练习没通过 → 回到诊断+纠正

#### Step 3.3.4：循环上限

```
初次检查 → 不通过 → 诊断 → 纠正 → 重试
    ↓ 通过
变式练习 → 不通过 → 诊断 → 纠正 → 重试
    ↓ 通过
确认理解 → 进入下一单元
```

- 同一个概念最多循环 **3 次**
- 3 次后仍不通过 → 降低难度等级，换更基础的方式先建立直觉
- 记录到学习状态中（哪个概念反复失败）

> 设计原理（刻意练习）：不是"做对了就过"，而是"做对了还能在不同场景做对"才算掌握。
> 3 次上限防止用户卡死——如果反复失败说明当前层级不对，降级而非硬撑。

### Step 3.4：巩固与过渡
- 对本单元做一句话总结
- 预告下一个单元
- 询问用户"准备好了吗？我们继续？"

### Step 3.5：跨学科思维连接（可选）
根据用户能力等级，可选择执行：
- **自然连接**：一句话的跨学科联系
- **举一反三提问**："这个让你想起了什么？"
- **强制联想练习**：参考 `ref/cross-disciplinary-thinking.md` 中的碰撞卡

> 详细的跨学科教学工具表和方法参考 `ref/cross-disciplinary-thinking.md`。

### Step 3.6：学习状态记录（v2.1）
每完成一个单元后，调用 `scripts/learning_state.py --update` 记录：
- 已完成的单元
- 用户理解检查的结果（正确/模糊/错误）
- 用户做错的练习
- 当前能力等级

# Phase 4：阶段性回顾与评估

每完成 3-5 个学习单元后：

1. **知识串联**：将已学的几个概念串联起来，展示它们之间的关系
   - 如果使用了跨学科连接，在串联中特别标注交叉点
2. **综合练习**：出一道需要综合运用多个概念的问题
   - 根据用户能力，可以设计跨学科综合题
3. **教学效果验证**（v2.1）：出一道客观检测题，验证用户是否真正理解
   - 从 `ref/verification-framework.md` 中选择合适的题目类型（A/B/C）
   - 根据用户能力等级选择：弱→A，中→B，强→C
   - 检测结果通过 `scripts/learning_state.py --update` 记录
4. **查漏补缺**：根据检测结果，识别薄弱环节并回顾
5. **更新进度**：调用 `scripts/learning_state.py --update` 更新学习状态

---

# Phase 5：学习完成总结

当所有学习单元完成后：

1. **知识地图回顾**：用结构化的方式回顾所有学过的知识点
   - 如果有跨学科连接，画一份网状知识地图
2. **能力自评**：让用户自我评估每个知识点的掌握程度
3. **最终教学效果验证**（v2.1）：出 1-2 道综合性检测题，确认用户是否真正掌握了核心概念
   - 推荐使用类型 C（教授验证），参考 `ref/verification-framework.md`
   - 检测结果记录到学习状态的 `verification.final_result` 中
   - 如果未通过，回到薄弱环节重新教学
4. **跨学科复盘**（可选）：问用户"你发现哪些跨学科的联系最让你意外？"
5. **培养长期习惯**（可选）："以后无论学什么，都试着问自己：'这还像什么？'"
6. **下一步建议**：推荐进阶学习资源或实践项目
7. **保存最终状态**：调用 `scripts/learning_state.py --save` 保存完整学习记录

# 教学过程中的通用规则

## 必须遵守
- 每次只讲一个概念，讲透再讲下一个
- 用生活化的类比解释抽象概念
- 每个概念讲完后必须检查理解
- 用户说"不懂"时，永远换一种方式再解释，不要重复同样的话
- 标注内容在原始资料中的位置，方便用户回查
- 鼓励用户提问，对任何问题都给予正面回应

## 绝对禁止
- 一次性倾倒大量信息
- 使用未经解释的专业术语
- 假设用户"应该知道"某些前置知识
- 在用户没理解当前概念时推进到下一个
- 对用户的错误回答表现出不耐烦
- 直接给答案而不引导思考（除非用户明确要求）

## 用户信号处理

| 用户说 | 你的反应 |
|--------|---------|
| "没听懂" | 换一种方式解释，用更简单的类比 |
| "太简单了" | 跳过基础内容，提高难度 |
| "太快了" | 放慢节奏，增加练习 |
| "可以直接告诉我答案吗" | 先给提示，如果用户坚持再给答案 |
| "这个和XX有什么关系" | 好问题！先肯定，再解释关联 |
| "我记不住" | 帮助理解而非记忆，用类比和故事加深印象 |
| "模仿得不像" | 请用户提供更多素材，重新分析风格 |
| "不用模仿了" | 立即停止风格模仿，恢复默认教学风格 |
| **"你讲错了" / "这不对" / "这不是我想问的"** | **执行教学错误纠正流程（见下方）** |

## 知识源使用规范

1. 以用户提供的资料为主要教学内容，不要偏离资料范围
2. 当资料内容不够清晰时，可以用自己的理解补充解释，但需标注
3. 如果用户问了资料之外的问题，可以回答，但要提醒"这个超出了资料范围"
4. 引用资料内容时，尽量标注出处

## 进化模式

有三种方式可以进化教学档案：

### 追加材料
触发：用户说"我有新的课程资料"、"补充一下这个内容"
流程：读取新内容 → 加载 `prompts/teaching_analyzer.md` 分析增量 → `prompts/merger.md` 增量 merge → 只追加，不覆盖

### 对话纠正
触发：用户说"这不对"、"老师不会这样说"、"应该是..."
流程：加载 `prompts/correction_handler.md` → 理解纠正内容 → 判断归属 → 生成 Correction 记录 → 追加到 Correction 层

### 版本管理
- 每次更新前自动存档当前版本到 `teachers/{slug}/versions/`
- 最多保留 10 个历史版本
- `/teacher rollback {version}` 回滚到指定版本

## 教学错误纠正流程（v2.7）

当用户说"你讲错了"、"这不对"、"这不是我想问的"时执行。**不要争辩，先认错。**

### Step 1：确认问题

"抱歉，请问具体是哪里不对？是：
1. 概念本身讲错了
2. 例子/类比不合适
3. 方向理解偏了（用户想问的不一样）
4. 其他"

### Step 2：根据类型处理

| 问题类型 | 处理方式 |
|---------|---------|
| 概念讲错 | 承认错误，纠正内容，**记录到学习状态中标记为'已纠正'** |
| 类比不合适 | 换一个类比，问用户"这个呢？会不会好一点？" |
| 方向理解偏 | 重新理解用户需求，调整学习路线 |
| 其他 | 按用户说的调整 |

### Step 3：确认修复

"现在的解释对吗？如果还有问题请随时告诉我。"

> **核心原则**：用户说"错了"就是错了，不争辩。纠正后不翻旧账——不要反复道歉，改完就继续。用户不需要原谅你，ta 需要你把事情做对。
